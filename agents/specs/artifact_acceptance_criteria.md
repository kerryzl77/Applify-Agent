# Artifact Acceptance Criteria

## Purpose

Define acceptance criteria for the user-visible outputs of one application run. These criteria are written for a seed-stage product: high trust, high usefulness, low configuration.

## Artifact Priority

1. Tailored resume
2. Cover letter
3. Outreach drafts

## Shared Criteria Across All Artifacts

- The artifact must reference the active job and candidate evidence from the same `application_run`.
- Claims must be grounded in known candidate inputs, not invented.
- The artifact must read like it was generated for this specific role, not a generic template.
- The artifact must be stored and reopenable without recomputing the run.
- The artifact must support regeneration without deleting unrelated artifacts.
- The artifact must be exportable or copyable in a format appropriate to its type.

## Tailored Resume Acceptance Criteria

### Product Goal

Create the strongest plausible resume for the target job. This is the primary artifact and the main reason the product exists.

### Functional Criteria

- A tailored resume is generated by default in every application run.
- The system uses the normalized job brief, not raw scraped text alone.
- The resume reflects the candidate’s existing experience, education, skills, and projects.
- The resume prioritizes the most relevant experience for the target role.
- The resume can be exported in at least one submission-friendly document format.
- The user can compare the latest tailored version against prior versions within the same run.

### Quality Criteria

- The top third of the resume makes clear why the candidate fits this role.
- Bullet points are materially tailored to the role’s requirements and keywords.
- Language is specific and credible for an early-career candidate; it does not overstate seniority.
- The resume improves relevance without stuffing keywords unnaturally.
- Formatting remains readable and professional after tailoring.

### Trust Criteria

- Every non-trivial claim can be traced to source profile/resume evidence.
- If evidence is weak for a key requirement, the system should de-emphasize or flag it rather than fabricate fit.
- If the profile is thin, the UI should warn that the resume may need closer review.

### Review Guidance

Human review is recommended.

Useful review tasks:

- factual correctness
- emphasis ordering
- tone calibration for conservative employers

Review should not require:

- manually approving each bullet before generation
- selecting among many template variants

## Cover Letter Acceptance Criteria

### Product Goal

Produce a concise, grounded cover letter that complements the tailored resume and gives the user a ready-to-send default.

### Functional Criteria

- A cover letter is generated by default in every application run.
- The cover letter uses the same job brief and evidence snapshot as the resume.
- The cover letter is stored as a first-class artifact in the run.
- The user can export or copy the cover letter without rerunning the full package.
- The user can request a focused revision to the cover letter only.

### Quality Criteria

- The letter clearly references the company and role.
- It explains fit using concrete experience, not generic enthusiasm.
- It adds value beyond repeating the resume line for line.
- It stays concise enough for realistic recruiter review.
- Tone is polished and professional without sounding inflated or robotic.

### Trust Criteria

- Company- or role-specific statements are grounded in the job brief or explicit user-provided information.
- The letter must not invent admiration, mission alignment, or domain expertise that is not supported.
- If the model lacks enough signal for a strong customized letter, the result should stay simple rather than hallucinate.

### Review Guidance

Human review is recommended, but should usually be fast.

Useful review tasks:

- adjust tone
- remove a weak paragraph
- add a company-specific sentence if the user knows more context

## Outreach Draft Acceptance Criteria

### Product Goal

Provide optional personalized outreach that extends the same application package, especially for recruiter or hiring manager contact.

### Functional Criteria

- Outreach drafts are generated as part of the same application run, not via a separate workflow.
- The initial set should include:
  - recruiter email
  - hiring manager email
  - LinkedIn note
- Each draft can be regenerated independently.
- Drafts can be copied immediately and, when configured, converted into Gmail drafts.
- Missing outreach drafts must not block package completion.

### Quality Criteria

- Each draft is specific to the job and candidate.
- Drafts are concise and scannable.
- The ask is clear and low-friction.
- The message does not read like mass outbound copy or generic networking spam.
- The content reflects early-career credibility and restraint.

### Trust Criteria

- The draft must not imply prior relationship unless supported.
- It must not claim deep product/company knowledge unless grounded.
- It must not promise attachments, portfolios, or referrals that do not exist.

### Review Guidance

Human review is optional by default.

Useful review tasks:

- confirm recipient relevance
- soften or sharpen tone
- add a specific sentence based on personal context

## Job Brief Acceptance Criteria

### Product Goal

Give the user a trustworthy summary of what the role actually asks for so they understand what the package is optimizing toward.

### Functional Criteria

- Every application run includes a normalized job brief.
- The brief appears before or during the run and remains available afterward.
- The brief highlights title, company, location, seniority, core responsibilities, top qualifications, and notable keywords.

### Quality Criteria

- The brief is concise enough to scan in under a minute.
- It separates hard requirements from inferred signals where possible.
- It helps the user understand why the resume was tailored a certain way.

### Review Guidance

Human review is not required for high-confidence extraction.

Review becomes useful when:

- extraction quality is visibly poor
- the posting is incomplete
- the role is unusually ambiguous

## Export And Persistence Criteria

- Resume and cover letter must remain available after the run completes.
- The latest accepted artifact version must be easy to identify.
- Export actions should be available from the application workspace, not hidden in secondary menus.
- Failure to export one artifact must not invalidate the run.

## Non-Goals

- large template libraries
- user-managed tone systems
- artifact-by-artifact setup before first run
- mandatory manual review gates for every artifact
